# 오답노트 (mistake-log)

목적: **같은 실수를 두 번 밟지 않는다.** 한 번 헛돈 함정(셸 인용·경로·Docker·마이그레이션·git)을 기록해 두고, 비슷한 작업 전에 먼저 본다. 면접 스토리(dev-log)와 달리 이건 **나(Claude)의 운영 치트시트** — 빠른 회피용.

## 노트 위치
- **이 스킬 폴더 안** `.claude/skills/mistake-log/오답노트.md` 에 누적한다(자기완결적 — 스킬과 노트가 한 곳).
- dev-log(`docs/dev-log/`)는 **시간순 개발 이력·면접 근거**. 오답노트는 **카테고리별 운영 함정·재발방지**. 역할이 다르니 중복 걱정 말 것 — 같은 사건이라도 dev-log엔 "증상→원인→해결→교훈" 스토리로, 오답노트엔 "다음에 이렇게 해라" 처방으로.

## 언제 기록하나 (트리거)
다음 중 하나라도 발생하면 즉시 한 줄이라도 남긴다:
- 명령/스크립트가 **실패**했다(컴파일·테스트·셸·Docker·SQL·git).
- **예상과 다르게** 동작했다(통과할 줄 알았는데 깨짐, 빈 결과, 멈춤/hang).
- **막다른 길**을 만나 접근을 바꿨다(첫 방법을 버리고 다른 방법으로).
- 사용자가 지적해 준 실수.

작은 오타·1회성 실수는 생략. **재현 가능하고 다음에 또 밟을 법한** 것만.

## 언제 참고하나 (이게 핵심 — 안 보면 의미 없음)
아래 작업을 **시작하기 전에** 오답노트.md의 해당 카테고리를 먼저 훑는다:
- **셸/PowerShell** 스크립트 작성 (인용·리다이렉트·cwd·native exe stderr)
- **Docker / MySQL** 컨테이너·`docker exec`·SQL 실행
- **Flyway / JPA** 마이그레이션 작성·`ddl-auto: validate` 런타임 검증
- **bootRun / 런타임 검증** 기동
- **git** 머지·푸시·스테이징

## 형식 (오답노트.md)
카테고리 헤딩 아래에 항목을 append. 한 항목 = 한 함정:

```markdown
## <카테고리: 셸 / Docker·MySQL / Flyway·JPA / Git / 런타임검증 / 기타>
### <짧은 제목> (YYYY-MM-DD)
- **증상**: 무엇이 어떻게 실패/오동작했나 (에러 메시지 핵심)
- **원인**: 왜
- **처방**: 다음엔 이렇게 한다 (복붙 가능한 명령/패턴이면 더 좋음)
```

규칙:
- **처방은 명령형·구체적으로.** "조심하자"가 아니라 "`docker exec`에 `-i` 빼고 `-e`로 실행".
- 같은 카테고리에 비슷한 항목이 쌓이면 묶어 정리.
- **고가치·자주 재발**하는 항목은 memory에도 미러링(매 세션 자동 로드되도록) — 예: [[persist-recurring-gotchas]] 원칙, [[project-windows-git-gotchas]].

## 기록 후
- 방금 막은 함정이 **이번에 실제로 시간을 잡아먹었다면**, dev-log의 "문제/해결"에도 스토리로 남길지 검토(면접 소재).
- 처방이 굳어지면 관련 스크립트/문서(`run.ps1`, README 등)에 반영해 애초에 안 밟게 만든다.
