## 셸 / PowerShell

### PowerShell cwd 드리프트 → 상대경로 bootRun 실패 (2026-06-09)
- **증상**: `backend\gradlew.bat -p backend bootRun`이 `... does not contain a Gradle build` 또는 `backend\backend\...` 경로로 실패. 로그 파일도 엉뚱한 곳에 생성.
- **원인**: Bash 툴에서 `cd backend` 한 게 하네스 cwd에 반영돼, 이후 PowerShell 호출의 cwd가 루트가 아니라 `backend`였음. 상대경로 `backend\...`가 `backend\backend\...`로 두 번 붙음.
- **처방**: 운영 스크립트는 **절대경로로 못박는다.** `$root="c:\gitProject\SideProjectWeb"; & "$root\backend\gradlew.bat" -p "$root\backend" bootRun *>&1 | Tee-Object "$root\backend\build\x.log"`. 로그 폴링/조회도 절대경로로.

### native exe(mysql/docker) stderr + `$ErrorActionPreference="Stop"` → 스크립트 중단 (2026-06-09)
- **증상**: `docker exec ... mysql ... 2>&1 | Where-Object {...}`가 `mysql: [Warning] Using a password ... insecure` 경고에서 `NativeCommandError`로 터지며 그 줄에서 스크립트 전체가 멈춤(이후 HTTP 호출 등 미실행).
- **원인**: PowerShell 5.1은 native exe의 stderr 한 줄을 ErrorRecord로 감싸고, `Stop`이면 그걸 종료 오류로 본다.
- **처방**: native exe 호출엔 **`2>$null`**(에러스트림을 네이티브 단에서 버림 → PowerShell이 안 감쌈). `Stop` 스크립트 안에서 docker/mysql은 별도 호출로 빼거나 `2>$null`. `2>&1`로 합치지 말 것.

### run.ps1은 `backend/`에서 실행 가정 (2026-06-09)
- **증상**: 루트에서 `& backend\run.ps1` → `gradlew bootRun`이 루트엔 gradle 빌드 없다며 실패.
- **원인**: `run.ps1`이 `gradlew.bat bootRun`을 cwd 기준으로 호출(프로젝트 디렉터리 `-p` 미지정).
- **처방**: `-p`로 프로젝트 디렉터리를 못박아 직접 기동. `.env` 로드 + `gradlew.bat -p <backend절대경로> bootRun`. (run.ps1을 쓰려면 cwd가 backend여야 함.)

## Docker / MySQL

### `docker exec -i ... -e "쿼리"` → stdin 대기로 hang (2026-06-09)
- **증상**: 단발 SQL 조회인데 명령이 안 끝나고 백그라운드로 매달림(빈 출력).
- **원인**: `-i`(interactive, stdin 유지)인데 파이프로 들어올 stdin이 없어 대기. `-e`는 실행 후 종료지만 `-i`가 세션을 붙잡음.
- **처방**: 단발 쿼리는 **`-i` 빼고** `docker exec commerce-mysql mysql -ucommerce -p"$pw" commerce -e "..." 2>$null`. (`-it`는 TTY 필요한 대화형에서만.)

### bootRun이 `Communications link failure`/`Connection refused` — MySQL 컨테이너 죽어 있음 (2026-06-10)
- **증상**: bootRun이 `com.mysql.cj.exceptions.CJCommunicationsException: Communications link failure`(또는 `Unable to determine Dialect`)로 기동 실패. 직전엔 잘 됐는데.
- **원인**: MySQL 컨테이너가 그새 **Exited(255)**. 세션 사이/시간 경과로 컨테이너가 죽어 localhost:3307 연결 거부.
- **처방**: bootRun **전에 컨테이너 살아있는지 확인**: `docker inspect -f "{{.State.Health.Status}}" commerce-mysql`(또는 `docker ps`). 죽어 있으면 `docker compose -f backend\docker-compose.yml up -d` → healthy 대기 후 기동. 데이터는 named volume이라 보존됨.

### 한글 LIKE 패턴이 PowerShell→docker stdin에서 깨짐 (이월 교훈)
- **증상**: `WHERE x LIKE '%한글%'`가 매칭 0(오탐). DB 값 자체는 정상 UTF-8.
- **원인**: PowerShell→docker stdin 전송 중 코드페이지로 한글이 깨짐.
- **처방**: 한글 비교는 LIKE 말고 **HEX/CHAR_LENGTH**로 검증. 콘솔에 깨져 보여도 DB/JSON은 UTF-8 정상일 때가 많음.

## Flyway / JPA (ddl-auto: validate)

### `double` 필드 ↔ MySQL `double` 컬럼은 validate 통과 (2026-06-09, V12 확인)
- **확인**: Java `double feeRate` ↔ Flyway `fee_rate double NOT NULL` → 기동(validate) 통과. 비율 같은 비-금액은 double 컬럼 OK. **돈(원)은 여전히 `bigint`(long).**
- **처방**: 비율/소수 필드는 `double NOT NULL` DDL로 두면 됨. 금액은 절대 double 금지.

### 손수 쓴 enum 컬럼은 값 집합을 **알파벳순**으로 (이월 교훈)
- **원인**: Hibernate가 `@Enumerated(STRING)`을 MySQL 네이티브 ENUM으로 매핑 → 마이그레이션의 `enum('A','B',...)` 값 순서가 다르면 validate 실패.
- **처방**: 마이그레이션 enum 값 나열을 **알파벳순**으로(엔티티 enum 상수도 알파벳순 유지). 컬럼 추가는 기존 행 위해 `NOT NULL DEFAULT '...'` 백필.

## 런타임 검증 (bootRun + HTTP)

### config-driven 동작은 env var로 런타임 데모 가능 (2026-06-09, 확인됨)
- **확인**: `@Value("${payment.unavailable-providers:}")` 같은 설정값은 bootRun 시 **env var로 주입된다**(Spring relaxed binding: `payment.unavailable-providers` ← `PAYMENT_UNAVAILABLE_PROVIDERS`). MPG-stretch 페일오버를 `$env:PAYMENT_UNAVAILABLE_PROVIDERS="KAKAOPAY"` 한 줄로 시연(yml 수정·재빌드 불필요).
- **처방**: 설정 분기(페일오버/플래그)를 런타임 데모할 땐 yml 건드리지 말고 **기동 직전 `$env:<UPPER_SNAKE>="..."`** 후 bootRun. `--args="--prop=val"`보다 PowerShell 인용이 깔끔.

### 검증 후 캐노니컬 복원은 필수 (상시)
- **처방**: 검증으로 만든 데이터는 끝나고 되돌린다 — `DELETE FROM mismatch/settlement_entry/payment/order_item/orders;`, `UPDATE product_option SET stock=50 WHERE id=1;`, `UPDATE member SET role='USER' WHERE email='buyer@commerce.com';`(검증 위해 ADMIN 승격했으면). 인메모리 PG 원장은 앱 종료로 리셋. 검증 끝나면 bootRun(TaskStop) 종료.
- **계정**: buyer@commerce.com/buyerpass1234(USER). ADMIN API는 buyer를 SQL로 ADMIN 승격 후 **재로그인**(role=JWT 클레임). 같은 member라 ADMIN 세션으로 본인 주문 생성·결제·취소까지 다 됨 → 한 세션으로 끝낼 수 있음.
