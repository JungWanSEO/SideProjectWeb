# 제품 기획 — 패션 셀렉트샵 (commerce-api)

> 작성 2026-06-10. 근거: `deep-research` 벤치마크(패션 셀렉트샵, 한국+글로벌) + 현재 코드베이스 gap.
> 목적: 포트폴리오용 패션 셀렉트샵의 **차별화 컨셉 + 기능 로드맵(MVP→확장)** 확정.

## 0. 한 줄 컨셉

> **"보이는 절반 + 보이지 않는 절반을 모두 구현한 입점 브랜드 셀렉트샵."**
> 대부분의 커머스 클론이 상품·장바구니·주문 CRUD에서 멈추는 지점 너머 — **다중 PG·정산·대사·트랜잭셔널 아웃박스** 같은 "운영/정산의 절반"을 이미 구현했다. 이 깊이를 차별화 축으로 두고, 고객 경험(패션 특화 탐색·리뷰·이미지)은 "진짜 쇼핑몰처럼 보이는" 수준까지 빠르게 채운다.

## 1. 벤치마크 핵심 (deep-research, 검증 통과분만)

패션 셀렉트샵 경쟁력은 4개 축으로 수렴 — 사실상 **무신사 1차 자료로 검증**된 결론(표본 편중은 §6 caveat).

| 축 | 검증된 사실 | 출처 | 백엔드 학습가치 |
|---|---|---|---|
| **패션 특화 필터** | 일반 커머스 필터를 넘어 색상(~50 팔레트)·패턴(20+)·핏(오버/레귤러)·**실측 수치 입력**(총장·어깨·가슴단면·소매). Baymard 5대 필수=가격·평점·색상·사이즈·브랜드 | 무신사 뉴스룸(2021-0930-03), Baymard | 中 (동적 쿼리·인덱싱) |
| **개인화 추천** | 스타일 택소노미(5→12), 행동 가중치(**구매>좋아요>조회**), "나를 위한 추천" | 무신사 테크블로그, 뉴스룸(2024-0920) | 高 (행동로그·배치·집계) |
| **UGC(스냅)** | 스타일링 콘텐츠 허브 + 사진 속 아이템→유사상품(FashionCLIP+객체탐지) 쇼퍼블 | 무신사 스냅, AWS 테크블로그 | 高 (임베딩·벡터검색) |
| **멀티스토어 큐레이션** | 홈을 6개 성장 카테고리(뷰티·스니커즈·부티크…)별 스토어로 개편 | 무신사 뉴스룸(2024-0813/1227) | 中 (큐레이션·편성) |
| **검색** | 텍스트=Elasticsearch(multi_match cross_fields + function_score BM25), 이미지=멀티모달 임베딩 | 무신사 테크블로그(MusE), AWS | 高 (ES·랭킹) |
| **재고/옵션(SKU)** | 사이즈·색상 SKU 동시성·오버셀링 방지(락·예약 패턴)가 백엔드 복잡도 1순위 | GeeksforGeeks, System Design Handbook | 高 (동시성) |

> ⚠️ 무신사 자사 발표 수치(전환율 3배 등)는 마케팅 지표라 "무신사 발표 기준"으로만 인용. 디자인/UX 정량 효과(360뷰 반품감소·썸네일·사진리뷰 선호율 등)는 **이번 리서치서 검증 실패** → 디자인은 일반 베스트프랙티스로만 다룬다.

## 2. 현재 commerce-api 위치

- **강점(이미 있음, 차별화 자산)**: 결제 다중 PG(AUTO 비용기반·페일오버)·**정산·대사·트랜잭셔널 아웃박스**·SKU 옵션(@Version 동시성)·JWT(httpOnly·OAuth2 prep)·QueryDSL 동적검색(백엔드)·카테고리/브랜드·어드민 콘솔.
- **약점(고객 경험 공백)**: 상품 이미지, 리뷰·평점, 패션 필터/검색 **UI**, 수량변경·배송지/주문서, 위시리스트, 추천, 푸터/SEO.

## 3. 차별화 전략

1. **운영·정산 깊이를 전면에 → "입점 브랜드 마켓플레이스"로 확장.** 기존 `brand` 도메인 + `settlement`/`reconciliation`을 재활용해 **브랜드(셀러)별 정산**으로 확장하면, 무신사식 "입점 브랜드 플랫폼"의 핵심 백엔드를 고유하게 시연. (대부분 클론이 절대 안 건드리는 영역)
2. **고객 경험은 '패션 필수기'만 빠르게.** 이미지·리뷰·패션 필터 3종으로 "진짜 패션몰처럼 보이는" 최소선을 만든다(과投자 금지).
3. **추천/검색은 단계적.** 추천=규칙기반 큐레이션→행동로그 기반, 검색=RDB/QueryDSL→Elasticsearch. 처음부터 ML/ES로 가지 않는다(§열린질문 의사결정).

## 4. 기능 로드맵 (MVP → 확장)

학습가치 = 백엔드 포트폴리오 어필도. 임팩트 = 고객이 체감하는 완성도.

### Phase 1 — "진짜 쇼핑몰로 보이게" (고객 필수기)
| 기능 | 백엔드 작업 | 학습가치 | 임팩트 |
|---|---|---|---|
| **상품 이미지(대표+갤러리)** | Product에 이미지 URL 필드 + 시드(실 placeholder) | 낮음 | ★★★ |
| **리뷰·평점 도메인** | 새 도메인: 구매자만 작성·평점 집계(평균/분포)·사진리뷰·신고 | **높음** | ★★★ |
| **패션 필터/정렬/검색 UI** | 색상·사이즈 필터를 QueryDSL에 추가 + FE 필터/검색바(백엔드 일부 기존) | 中 | ★★ |
| **체크아웃 완성** | 수량 변경·배송지(주소록)·주문서 | 中 | ★★ |

### Phase 2 — 차별화·백엔드 심화
| 기능 | 백엔드 작업 | 학습가치 |
|---|---|---|
| **입점 브랜드(셀러)별 정산** | brand→seller 확장, 주문 분할정산·셀러 정산서(기존 settlement 확장) | **매우 높음(차별화)** |
| **쿠폰/프로모션 엔진** | 발급·중복·최소금액·한도·만료, 주문 적용 | **높음** |
| **위시리스트/찜** | 새 도메인(찜·좋아요), 인기도 집계 연계 | 낮음~中 |
| **개인화 추천(규칙→행동)** | 조회/좋아요/구매 행동로그 → 큐레이션/추천 배치 | 高 |

### Phase 3 — 고급(여력 시)
| 검색 Elasticsearch 전환 | 동의어·오타·랭킹·자동완성 | 高 |
| UGC/코디(스냅) | 스타일 게시물 + 상품 태깅(쇼퍼블) | 中 |
| 재고 동시성 심화 | 예약(reserved) 패턴·플래시세일·재입고 알림 | 高 |

## 5. 백엔드 역량 매핑 (면접 어필)

| 역량 | 해당 기능 | 어필 포인트 |
|---|---|---|
| 동시성·일관성 | SKU 재고(@Version)·예약 패턴·플래시세일 | 오버셀링 0 증명(이미 있음→심화) |
| 정산·정합성 | 다중 PG·정산·대사·셀러 정산 | "매출≠결제액", 두 출처 대조(이미 강점) |
| 비동기·신뢰성 | 아웃박스·재입고/리뷰 알림 | 트랜잭셔널 아웃박스(이미 있음) |
| 집계·도메인 모델 | 리뷰 평점 집계·인기도·쿠폰 엔진 | 규칙 엔진·집계 설계 |
| 검색·랭킹 | QueryDSL→Elasticsearch | 동적쿼리→검색엔진 전환 스토리 |

## 6. 추천 시작점 & 열린 질문

**추천 순서(Phase 1):** ① 상품 이미지(비주얼 즉효) → ② 리뷰·평점(새 백엔드 도메인, 어필 큼) → ③ 패션 필터/검색 UI(기존 백엔드 활용 가성비). 그 후 Phase 2의 **셀러별 정산**을 차별화 하이라이트로.

**열린 질문(다음 리서치 후보):**
- 29CM·SSENSE·Zara 등 **경쟁사별 고유 기능/UX** (이번엔 무신사 편중으로 미검증)
- PDP 전환 UX(스티키 구매바·사이즈 추천)·모바일 패턴의 **신뢰 가능한 1차 근거**
- 쿠폰 엔진·리뷰 집계·재입고 알림의 백엔드 구현 패턴 비교
- 검색=RDB풀텍스트 vs ES, 추천=규칙 vs 임베딩의 **단계 전환 기준**
