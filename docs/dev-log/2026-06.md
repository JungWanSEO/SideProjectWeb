## 2026-06-02 (2일차) — 인증 Phase 2 (Refresh 토큰)

### 분석 (사용자 질문: BCrypt + rainbow table / salt / IV / DB 변경?)
- **BCrypt는 패스워드마다 랜덤 salt를 내장**(`$2a$10$<22자 salt><해시>`)하므로 rainbow table이 이미 방어됨. **별도 salt 컬럼·DB 변경 불필요.**
- **IV는 암호화(양방향, AES) 개념** — 단방향 해싱인 비밀번호엔 해당 없음.
- 정당한 "추가"는 salt/IV가 아니라 **pepper**(앱 시크릿, DB 밖) 또는 cost factor 상향. (이번엔 미적용)
- 실제 DB 해시 확인으로 증명. 발견된 **평문 alice 레코드(BCrypt 이전 생성)**는 삭제 후 ADMIN으로 재생성.

### 추가
- **RefreshToken** 엔티티(memberId unique, token) + Repository → **DB 저장**(회전·폐기 통제). ※ 비밀번호와 달리 refresh 저장은 정당한 DB 추가.
- TokenResponse에 `refreshToken` 추가, `TokenRefreshRequest` 추가.
- AuthService: 로그인 시 refresh 저장, **`refresh()`**(검증→저장본 일치 확인→새 토큰 발급+회전).
- AuthController: `POST /api/auth/refresh`.
- 테스트 추가(로그인/재발급/회전/재사용거부) → **전체 58개 통과.**
- 런타임 검증: 회전(새≠옛) / 옛 refresh 재사용 401 / 새 refresh 200 / ADMIN 계정으로 상품등록 201.

### 결정
- **리프레시 회전(rotation)**: 재발급 때마다 새 refresh 발급+저장본 교체. 이전 토큰은 무효 → 탈취 재사용 방어.
- 저장본과 불일치하는 refresh는 거부(재사용/폐기 탐지).

### 문제 / 해결
- **[증상]** 회전했는데 새 refresh가 옛 것과 동일하고, 옛 토큰 재사용이 막히지 않음(런타임에서만 발견, 단위테스트는 mock이라 통과).
  - **[원인]** JWT `iat`/`exp`는 **초 단위**. 로그인과 즉시 refresh가 같은 초에 일어나면 (sub+iat+exp 동일 →) 토큰 문자열이 byte-identical.
  - **[해결]** 토큰에 **고유 `jti`(UUID)** 클레임 추가 → 매 발급마다 다른 토큰. 회전·재사용 탐지 정상화.
  - **[교훈]** JWT 시간 클레임은 초 단위라 유일성 보장 안 됨. 유일성이 필요하면 jti를 넣어야. 단위테스트(mock)로는 못 잡고 런타임 검증에서 드러남.

- **충돌 처리 = 재시도(spring-retry)**: 초과판매는 @Version이 막고, 재시도는 정당한 주문이 충돌로 실패하지 않게 하는 UX 보강.

- **동시성 테스트는 불변식으로 단정**: maxAttempts(3) 고정이라 "정확히 N개 성공"은 경합에 따라 흔들릴 수 있음 → ①성공≤재고 ②재고≥0 ③성공=차감량(lost update 없음)으로 비flaky 검증.

### 문제 / 해결
- **[설계 함정]** `@Retryable`+`@Transactional`을 같은 메서드에 두면 어드바이스 순서에 따라 "같은(이미 깨진) 트랜잭션 재시도"가 될 수 있음.
  - **[해결]** 트랜잭션 워커(OrderProcessor)를 별도 빈으로 분리 → 재시도가 항상 새 트랜잭션을 연다.
  - **[교훈]** 낙관적 락 재시도는 반드시 트랜잭션 **바깥**에서. "retry wraps a fresh transaction".

- **목록도 `OrderResponse` 재사용 + 배치페치** (요약 DTO 대신): 코드·패턴 재사용 극대화 + N+1 정석 대응 학습. 실제 커머스식 요약 DTO("대표상품 외 N건")는 후속 선택지로 남김.
- **N+1 대응 = `hibernate.default_batch_fetch_size: 100`** (운영·테스트 yml 양쪽): 목록 N건의 지연 `orderItems` 로딩을 `where order_id in (?, …)` 한두 방으로 묶음. **컬렉션 fetch join + 페이징은 금지**(Hibernate가 메모리 페이징 → 위험). 전역 설정이라 다른 지연 로딩도 함께 개선.
- **보안 변경 불필요**: `/api/orders`는 SecurityConfig의 `anyRequest().authenticated()`에 걸려 이미 인증 필요. (상품 목록은 permitAll이라 반대 케이스였음.)

### 문제 / 해결
- (특이사항 없음 — 컴파일·테스트 1차 통과.)
- **(후속 TODO 발견)** 기존 `GET /api/orders/{id}`는 소유자 검증이 없어 인증된 누구나 임의 주문 ID를 조회 가능(IDOR 소지). 이번 목록은 memberId 필터라 안전. 단건 소유자 체크는 후속 과제로 기록.

## 2026-06-03 (3일차) — 주문 접근 권한(IDOR) 보강

### 문제 / 해결 — IDOR (남의 주문 조회·취소)
- **[증상]** `GET /api/orders/{id}`·`POST /api/orders/{id}/cancel`이 소유자 검증 없이 주문 ID만으로 동작 → 인증된 사용자가 **남의 주문을 조회·취소**(재고까지 복원) 가능. (IDOR: Insecure Direct Object Reference)
- **[해결]** 본인 주문이거나 ADMIN일 때만 허용, 아니면 **403**:
  - `SecurityUtil.isAdmin()` 추가(권한 `ROLE_ADMIN` 보유 여부).
  - 컨트롤러가 `getCurrentMemberId()`+`isAdmin()`을 서비스에 전달(기존 패턴 — 서비스는 SecurityContext 비의존).
  - `OrderService.getOrder/cancel(id, requesterId, admin)`로 시그니처 변경 + 공통 `findOwnedOrder`에서 소유자/ADMIN 검증.
- 테스트 5개 추가: Service 4(본인 조회 성공 · 남의 조회 403 · ADMIN은 남의 것도 조회 · 남의 취소 403+상태 불변) / Controller 1(403 응답). 기존 호출부(조회/취소) 시그니처도 갱신. **전체 73개 통과**(68→73).

### 결정
- **응답 코드 = 403 Forbidden** (인증은 됐으나 권한 없음). 존재 자체를 숨기는 **404 하드닝**도 가능하나, 의미 명확성을 우선해 403 채택(전환은 한 줄).
- **권한 판정은 서비스 계층**(컨트롤러가 값만 전달): 단위 테스트에서 SecurityContext 없이 (requesterId, admin)으로 직접 검증 가능 → 테스트 용이.
- 통합 시나리오(USER가 본인 주문 조회·취소)는 소유자라 영향 없음 — 그대로 통과.

## 2026-06-03 (3일차) — 상품 옵션(사이즈) 설계 합의 (구현 전)

### 결정 (사용자와 합의 — 패션 셀렉트샵식 분석 후)
- **옵션 축 = 단일(사이즈).** 색상은 별도 상품(패션 셀렉트샵 정체성). 다축(사이즈×색상 SKU 매트릭스)은 일반 이커머스식 — 조합 폭발·복잡도로 제외.
- **재고는 옵션별**: `ProductOption.stock`. 재고 메서드(`decreaseStock`/`increaseStock`)와 `@Version` 낙관적 락이 Product→ProductOption으로 이동(사이즈=SKU). **Product.stock 제거**(단일 출처=옵션).

- **주문/장바구니는 optionId 참조** + size 스냅샷. 주문 요청 `items:[{optionId, quantity}]`.
- **재고 차감 = Product 루트 경유**: optionId로 Product 애그리거트 로드(이름·가격 스냅샷에 어차피 필요) → 내부 옵션 차감(옵션 `@Version`으로 충돌 감지). 한 번 로드로 해결 + 애그리거트 경계 유지.
- **기존 데이터 마이그레이션**: 상품마다 옵션 1개("FREE", stock=기존 Product.stock) 시드.

### 구현 페이즈 (한 번에 X)
- **P1 Product+Option**: ProductOption 엔티티·재고/@Version 이동, 상품 등록/조회에 옵션, 사이즈별 품절, 옵션 동시성 테스트.

- **P3 Cart**: cartItem.optionId + enrich.

- 패션 셀렉트샵: 단일 축(사이즈)·옵션별 재고·가격 동일·색상=별도상품 → 단순.

## 2026-06-03 (3일차) — 상품 옵션 P1: 재고 모델 옵션 이동 + 주문 컷오버 (통째)

> 합의 시 P1=Product+Option / P2=Order로 나눴으나, **"재고를 옵션으로"와 "주문이 옵션을 가리킴"이 결합**(Product.stock 제거 시 주문이 깨짐)이라 사용자 확인 후 **한 번에 컷오버**(Y안). 단일 재고 출처로 깔끔하게.

### 추가 / 변경
- **`ProductOption`**(Product 애그리거트 내부 `@OneToMany`, cascade/orphanRemoval): id·size·stock·`@Version`·`decreaseStock`/`increaseStock`/`isSoldOut`. **재고·@Version·재고메서드가 Product→Option으로 이동**, **Product.stock 제거**.
- **Product**: `@OneToMany options` + `addOption`·`decreaseStock(optionId,qty)`·`increaseStock(optionId,qty)`·`optionSize(optionId)`(루트 경유 위임).
- **ProductRepository.findByOptionId**(`@Query` join): optionId로 Product 애그리거트 로드(루트 경유 차감용).

## 2026-06-03 (3일차) — 상품 옵션 P1 런타임 검증 + 레거시 컬럼 정리 (MySQL 마이그레이션)

> P1은 그동안 H2 테스트(88개)만 통과했고 실제 MySQL에서 한 번도 안 돌렸음. 운영형 DB에서 옵션 주문 흐름을 검증하고, 검증 중 드러난 스키마 부채를 정리. **앱 코드 변경 없음 — DB 마이그레이션 + 런타임 검증만.**

### 검증 (런타임, PASS)
- **DB 컷오버 자동 반영 확인**: 이 MySQL은 "상품목록 검증(+랜섬복구)" 시점 스키마에 멈춰 있었음(category/brand/product_option 테이블 없음, product에 category_id/brand_id 없음, order_item에 option_id/size 없음). `bootRun` 시 **Hibernate ddl-auto:update가 그 사이 누적된 변경(2b·검색·옵션 P1)을 한 번에 반영** — 세 테이블 생성 + product에 category_id/brand_id 추가 + order_item에 option_id/size 추가. (= 2b 이후 줄곧 MySQL 미적용 상태였음을 확인)
- **마이그레이션(FREE 옵션 시드)**: 기존 7상품은 옵션이 없어 주문 불가 → 상품마다 옵션 1개 시드. **시드값 출처 = 잔존하던 레거시 `product.stock` 컬럼**(50/30/20/100/10/80/15). `INSERT ... SELECT id,'FREE',stock,0,NOW(6),NOW(6) FROM product WHERE id NOT IN (SELECT product_id FROM product_option)`.
- **옵션 주문 흐름(기존 상품 P06-Tee, option 6)**: 상세 GET에 옵션 노출(id6·FREE·stock80·soldOut false) → `POST /api/orders {items:[{optionId:6,quantity:3}]}` → **201, total 120000, size "FREE" 스냅샷** → 재고 80→77 → `GET /api/orders/{id}` ORDERED → `cancel` CANCELLED → 재고 77→80 **복원**. 전부 PASS.
- **음성 경로**: 재고부족(option5 stock10에 qty11) → **409**, 없는 옵션(optionId 999) → **404**.
- **신규 상품 옵션 등록 경로(레거시 컬럼 정리 후 첫 런타임 검증)**: `POST /api/products`(옵션 M:5/L:3) → **201**, M옵션 주문 qty2 → 재고 5→3. (create-with-options 경로도 그동안 H2로만 검증됐었음)

### 🔴 문제 / 해결 — 레거시 `product.stock` 컬럼이 신규 상품 등록을 깨뜨림
- **[증상]** 앱과 동일하게 `stock`을 생략한 INSERT가 **`ERROR 1364 (HY000): Field 'stock' doesn't have a default value`**로 실패 → `POST /api/products`가 이 MySQL에서 500 날 상태였음.
- **[원인]** P1에서 `Product.stock`을 엔티티에서 제거했지만 **ddl-auto:update는 컬럼을 자동 삭제하지 않음**. 레거시 `stock` 컬럼이 `NOT NULL`(기본값 없음)로 잔존 → 앱의 INSERT(stock 미포함)를 MySQL strict mode가 거부. (`product.version`도 `@Version`이 ProductOption으로 내려가며 죽은 컬럼으로 잔존 — nullable이라 insert는 안 깨뜨림)
- **[원인2 — H2가 못 잡은 이유]** 테스트 DB(H2)는 매번 **현재 엔티티로 새로 생성(create-drop)** → 레거시 stock 컬럼이 애초에 없어 이 불일치를 구조적으로 못 잡음. ⇒ **P1 dev-log의 "(product.stock 컬럼 … 무해하게 잔존)"은 사실과 달랐다.** 런타임 검증의 가치가 정확히 여기서 드러남.
- **[해결]** 시드 완료(stock값 보존) 후 `ALTER TABLE product DROP COLUMN stock, DROP COLUMN version`(사용자 합의). 이후 신규 상품 등록 201 정상화 확인. product 테이블이 현재 엔티티와 1:1 일치(id·created_at·updated_at·description·name·price·status·brand_id·category_id).
- **[교훈]** 엔티티 필드를 지워도 `ddl update`는 컬럼을 안 지운다 → **NOT NULL 잔존 컬럼은 insert를 깨뜨릴 수 있다.** H2(create-drop)는 "기존 스키마와의 불일치"를 못 잡으니, 운영형 DB의 런타임 검증이 필수. (운영에선 Flyway/Liquibase로 컬럼 제거를 명시 관리 — ddl update는 로컬 학습 한정.)

### 정리 / 환경
- **검증은 새 USER `buyer@commerce.com`/`buyerpass1234` 가입으로 진행**(기존 admin BCrypt 비번 미상) — 실제 구매자 흐름 미러링. 신규 상품 등록 검증 시 buyer를 SQL로 ADMIN 임시 승격 → 검증 후 USER 원복(role은 JWT 클레임이라 승격/원복마다 재로그인).

## 2026-06-04 (4일차) — 인증 Phase 3: JWT를 httpOnly 쿠키로 (+ 로그인 UI)

> 기존 인증은 토큰을 응답 body로 주고 `Authorization: Bearer` 헤더로 받았음(FE라면 localStorage/메모리 보관 필요). FE 착수에 맞춰 **토큰 보관 위치 전략**을 다시 정하고 httpOnly 쿠키로 전환.

### 결정 (사용자 합의 — 토큰 보관 전략 / "어디에·왜")
- **JWT(access+refresh)를 httpOnly 쿠키에 보관.** 위협모델 트레이드오프: **XSS**(악성 JS가 토큰 탈취) vs **CSRF**(브라우저가 위조 요청에 쿠키 자동첨부). localStorage=XSS 취약 / httpOnly 쿠키=JS가 못 읽어 XSS 안전하지만 CSRF 노출.
- 채택: **httpOnly로 XSS 차단 + `SameSite=Lax`로 CSRF 차단**. **access·refresh 둘 다 httpOnly 쿠키**(JS가 토큰을 전혀 안 만짐 — 가장 순수). 근거: 포트폴리오에서 "토큰을 어디 두고 왜"가 핵심 평가 포인트 + 보안 민감(랜섬 경험).
- 운영 TODO: https 시 `secure=true`, 프론트/API 도메인이 갈리면 `SameSite=None`+CSRF 토큰.

### 추가/변경 (백엔드)
- **`AuthCookieManager`**: access/refresh 쿠키 생성·삭제(httpOnly·SameSite=Lax·secure(dev=false)·path=`/`·maxAge=토큰 유효기간 access 30분/refresh 14일).
- **로그인/재발급 → 토큰을 body 대신 `Set-Cookie`로**, 응답 body는 유저정보(`MemberResponse`). `AuthService.login/refresh` → `AuthResult`(토큰문자열+user), `refresh(String)`는 쿠키에서 받은 토큰을 인자로.
- **`JwtAuthenticationFilter`**: Authorization 헤더 → 없으면 **`access_token` 쿠키**에서 토큰 추출(헤더 폴백 유지 → Swagger·API클라이언트·테스트 호환).
- **신설 `GET /api/auth/me`**(현재 유저 — httpOnly라 FE가 토큰을 못 읽으니 "누구인지" 확인용 필수 짝꿍), **`POST /api/auth/logout`**(refresh 폐기 + 쿠키 삭제, 멱등).
- `SecurityConfig`: `/api/auth/login·refresh·logout` permitAll, `/api/auth/me`는 인증 필요. `TokenResponse`·`TokenRefreshRequest` 제거.
- 테스트: AuthService(AuthResult·logout 폐기검증), AuthController(쿠키 set/clear·/me·**토큰 body 부재 검증**), CommerceScenario(**쿠키 캡처→재전송 E2E**). **전체 90개 통과**(88→90, 로그아웃·/me 테스트 추가).

### 추가 (프론트엔드)
- `lib/api.ts`: **`credentials:"include"`**(httpOnly 쿠키 송수신 — CORS allowCredentials와 짝) + `apiPost`.
- `lib/auth.tsx`: `AuthProvider`/`useAuth` — 최초 `GET /me`로 로그인상태 판단, `login()`/`logout()`.
- `components/Header.tsx`: 로그인상태별 네비(로그인 ↔ "유저님+로그아웃"). `layout`에 `AuthProvider`+`Header`.

## 2026-06-08 (7일차) — 정산(Settlement) P1: 정산 기록 도메인

> 결제 심화의 첫 단계. 결제(승인)는 한 점이지만 **정산은 며칠 뒤 수수료를 떼고 입금되는 별도 시점**이다. "매출 ≠ 결제액"을 1급 필드로 모델링. 개념 근거는 `docs/payment-modern-architecture.md §3`. (대사(reconciliation)는 P2.)

### 추가 (settlement 도메인 신설 — add-domain 컨벤션)
- **`SettlementEntry`(entity) + `SettlementStatus`(enum SCHEDULED→PAID_OUT)**: `paymentId·orderId`(ID 참조), **`pgTransactionId`(대사 조인 키)**, `grossAmount`(결제액)·`fee`(수수료)·**`netAmount`(실입금 = gross − fee)**, `settledDate`(T+N). net은 엔티티가 스스로 계산(파생값 일관성).
- **`SettlementPolicy`**: 수수료율(모의 2.5% 고정)·입금지연(T+2)을 한 곳에. 실무에선 결제수단·카드사·상품별로 달라 설정/요율표로 분리(§3.6).
- **`SettlementService`**: `run()`=정산 배치(PAID 결제 중 미정산 건을 모아 SCHEDULED 생성, 합계 요약 반환) / `getSettlements(page)` / `payout(id)`(SCHEDULED→PAID_OUT).
- **`SettlementController`** (`/api/settlements`, 전부 ADMIN): `POST /run` · `GET` · `POST /{id}/payout`. DTO `SettlementResponse`·`SettlementRunResponse`.
- **payment 측 보강**: `PaymentRepository.findByStatus` + **`PaymentService.getPaidPayments()`(DTO 반환)** — settlement→payment 의존을 **서비스+DTO로만** 노출(엔티티/리포지토리 직접 안 넘김 = 도메인 경계 유지).
- **SecurityConfig**: `/api/settlements/**` → `hasRole("ADMIN")`.
- **Flyway `V5__add_settlement.sql`**: `settlement_entry` 테이블. `payment_id` **UNIQUE**(한 결제당 정산 1건 → 배치 멱등), status enum 값은 알파벳순(PAID_OUT, SCHEDULED → validate 통과).
- 테스트: **SettlementServiceTest +5**(배치 수수료·실입금 계산 / 멱등 스킵 / payout / 404 / 409) → **전체 112개 통과**(107→112).

### 결정
- **정산은 별도 도메인**(payment에 안 섞음): 결제는 "거래 승인됐나", 정산은 "며칠 뒤 얼마 입금됐나" — 생명주기·관심사가 다름(DDD 경계). Payment에 `fee`·`settledAt`을 욱여넣으면 결제 도메인이 정산 걱정까지 오염.
- **생성 시점 = 배치 스캔**(사용자 선택, vs 결제 PAID 시 동기 생성): ① 실무의 "T+N 일배치"를 모델링 → "정산=별도 시점" 교훈 유지 ② 결제 쓰기경로(PaymentService.pay) 안 건드림 ③ settlement→payment 의존 방향이 자연스러움(역방향이면 순환).
- **매출 ≠ 결제액**: gross/fee/net 세 필드로 분리해 명시. 정산 모델이 수수료를 1급으로 다룸.
- **멱등성 = `existsByPaymentId` + DB `payment_id` UNIQUE**: 배치를 여러 번 돌려도 중복 정산 없음(결제의 idempotencyKey와 같은 사고방식, 다른 레이어).

### 런타임 검증 (MySQL, 2026-06-08)
- **Flyway V5 + validate 통과**: `docker compose up -d` → `./run.ps1`. 앱 기동 성공 자체가 "V5 SQL ↔ 엔티티 일치"의 증거. `flyway_schema_history` V5 `success=1`, `settlement_entry` 구조(payment_id UNIQUE·status enum('PAID_OUT','SCHEDULED')) DB로 확인.
- **HTTP end-to-end(쿠키 세션)**: buyer로 PAID 결제 2건(40,000·10,000) 생성 → ① **ADMIN 가드**: buyer가 `POST /run` 호출 시 **403** ② **정산 배치** `POST /run`: created=2·gross 50,000·**fee 1,250·net 48,750**(2.5%) ③ **매출 ≠ 결제액** 항목별 확인(40,000→net 39,000 / 10,000→net 9,750) ④ **settledDate = T+2**(2026-06-10) ⑤ **payout** SCHEDULED→PAID_OUT ⑥ **멱등 재실행** created=0. 검증 후 캐노니컬(재고 Cap 50·Tee 80, orders/payments/settlements 0) 복원.

## 2026-06-08 (7일차) — 정산(Settlement) P3: 불일치 해소(resolve) 워크플로

> 대사의 "예외 큐"를 **검출 → 처리**까지 완성. 사람이 불일치를 resolve(상계·보정)/ignore(오탐)하고, 재대사는 그 결정을 존중해 처리된 건을 다시 열지 않는다.

### 추가
- **`Mismatch` 생명주기**: `MismatchStatus`(OPEN→RESOLVED/IGNORED, 알파벳순) + `resolutionNote`. 엔티티에 `resolve(note)`/`ignore(note)`(OPEN에서만, 아니면 409 — 상태머신 가드), 생성 시 OPEN.
- **`reconcile()` 동작 변경**: `deleteAllInBatch`→**`deleteByStatus(OPEN)`**(처리된 건 보존). 재대사 전 `findByStatusIn(RESOLVED,IGNORED)`로 **이미 처리된 거래키 수집** → 그 키의 불일치는 다시 OPEN으로 안 만들고 **`alreadyHandled`**로만 집계. `ReconciliationResult`에 `alreadyHandled` 추가.
- **API**(ADMIN): `POST /mismatches/{id}/resolve`·`/ignore`(body `MismatchResolveRequest{note}` 선택), `GET /mismatches?status=OPEN`(상태 필터). `MismatchResponse`에 status·resolutionNote 추가.
- **`MismatchRepository`**: `findByStatusIn`·`deleteByStatus`·`findByStatus(pageable)`. **Flyway V7**(mismatch에 status·resolution_note ADD, status DEFAULT 'OPEN'로 안전 ALTER).
- 테스트: ReconciliationServiceTest **+5**(재대사 스킵·resolve·ignore·404·409) + matched 검증 갱신 → **전체 123개**(118→123).

### 결정
- **재대사는 OPEN만 스냅샷·처리된 건 보존**: 예외 큐의 기본 동작 — resolve/ignore한 거래키는 다시 OPEN으로 안 뜸(사람 결정 존중). 단순화: **재오픈 정책 생략**(한번 처리하면 데이터가 여전히 어긋나도 다시 안 뜸 — 실무는 더 복잡).
- **resolve vs ignore 분리**: 둘 다 OPEN→종료지만 의미 구분(RESOLVED=상계/보정함, IGNORED=오탐/허용). 엔드포인트도 분리해 자기설명적.
- **note는 선택(body optional)**: 사유 없이도 처리 가능(`@RequestBody(required=false)`). 운영 편의.

### 런타임 검증 (MySQL, 2026-06-08)
- **Flyway V7 + validate 통과**: V7 `success=1`, `mismatch.status`(enum default OPEN)·`resolution_note` 컬럼 확인.
- **생명주기 end-to-end**: B(Cap) 결제→정산→환불(STATUS_MISMATCH), C(Socks) 정산후결제(MISSING_IN_OURS) → **run#1**: OPEN 2건(total=2·alreadyHandled=0) → STATUS_MISMATCH **resolve**·MISSING_IN_OURS **ignore** → **run#2**: **total=0·alreadyHandled=2**(처리한 2건 다시 안 열림) → status 필터 OPEN=0·RESOLVED=1·IGNORED=1, resolutionNote 보존 확인. 검증 후 캐노니컬 복원.

## 2026-06-08 (7일차) — 이벤트·아웃박스 설계 + P1 구현

> 결제 완료를 안정적으로 이벤트 발행. **dual-write 문제**(상태변경 DB ↔ 이벤트 MQ를 한 트랜잭션에 못 묶음)를 **트랜잭셔널 아웃박스**로 해소. `PaymentService` 주석에 적어둔 갭 + architecture.md §13.5 예고를 실제로 구현.

### 설계 (먼저, 코드 X)
- **`docs/event-outbox-design.md`** 신설: dual-write 문제 · 패턴(같은 tx INSERT + 폴러 at-least-once) · 대안 비교(`@TransactionalEventListener`는 durability 없어 부족, 2PC 회피) · outbox 스키마 · 시퀀스 · 신뢰성 결정(멱등·재시도/데드레터·순서) · 확장(MQ/CDC) · §9 결정 테이블. .NET 비유(MediatR 도메인이벤트 + MassTransit/EF 아웃박스).

## 2026-06-08 (7일차) — 이벤트·아웃박스 P2a: 지수 백오프 + SKIP LOCKED

> P1 폴러의 두 약점을 보강. ① 실패 이벤트를 **2초마다 무한 재시도**(PG 장애 시 헛때림) → **지수 백오프**. ② 폴러를 여러 개 띄우면 **같은 행 중복 발행** 위험 → **`FOR UPDATE SKIP LOCKED`** 행 잠금. 설계 노트 §5·§7이 P2로 예약해둔 항목. **인프라 0**(DB 컬럼 1 + 쿼리 2 + 정책). 실제 RabbitMQ 어댑터는 P2b(스택 추가)로 분리.

### 추가/변경 — 지수 백오프
- **`OutboxEvent.nextAttemptAt`**(nullable) 추가: "다음 재시도 가능 시각". 신규 이벤트는 `null`=즉시 대상, 실패할수록 미래로 밀림.
- **`recordFailure(error, maxRetries, nextAttemptAt)`**: PENDING으로 남을 때만 `nextAttemptAt` 반영(FAILED=데드레터면 무의미). 정책은 **프로세서가 소유**(기존 `maxRetries` 넘기던 패턴 그대로) — `OutboxProcessor.nextAttemptAfter(retryCount)` = `now + min(base·2^retryCount, max)`. base/max는 `@Value`(`outbox.relay.backoff-base-ms:2000`/`-max-ms:60000`) → 대기 **2s→4s→8s→16s**(상한 cap). retryCount는 증가 전 값이라 실패 1·2·3·4회에 그대로 매핑.
- **폴러 후보 스캔에 백오프 필터**: `findDispatchable(now)` = `WHERE status='PENDING' AND (next_attempt_at IS NULL OR next_attempt_at <= now) ORDER BY id LIMIT 100`.

### 추가/변경 — SKIP LOCKED (폴러 스케일아웃 안전)
- 패턴 = **"싼 후보 스캔 + 행별 잠금 클레임"**: `findDispatchable`(잠금 없음)으로 후보를 훑고, `OutboxProcessor.publish` 안에서 재조회를 **`findPendingForUpdate(id)` = `SELECT … WHERE id=? AND status='PENDING' FOR UPDATE SKIP LOCKED`**로 승격. 다른 폴러가 이미 그 행을 처리 중(잠금 보유)이면 **빈 결과 → 즉시 스킵**(대기 없음). 잠금은 publish 트랜잭션 동안 유지 → 이벤트별 트랜잭션 구조 그대로 보존. (P1의 `findById`+`status!=PENDING` 가드를 잠금 클레임으로 승격 — 변경 최소.)
- **Flyway V10**(`add_outbox_backoff`): `next_attempt_at datetime(6) NULL` 추가 + 조회 인덱스 `idx_outbox_status` → **`idx_outbox_dispatch(status, next_attempt_at)`** 복합으로 교체.
- 테스트: `OutboxProcessorTest` publish 스텁 `findById`→`findPendingForUpdate`, 잠금 클레임 비면 스킵, **백오프 지수 증가 검증**(실패 거듭→`nextAttemptAt` 간격 2s→4s→8s) → **전체 131개**(130→131).

### 결정
- **백오프 정책은 프로세서, 엔티티는 저장만**: `maxRetries`처럼 `nextAttemptAt`도 계산해 넘김 → base/max를 설정값으로 빼고 단위테스트도 용이(`@Value` 필드는 `ReflectionTestUtils`로 주입).
- **SKIP LOCKED는 native 쿼리**(JPQL 표현 불가): `FOR UPDATE SKIP LOCKED`. **H2(테스트)는 미지원**이라 단위테스트(Mockito·목)는 이 SQL을 안 타고, 실제 경로는 **MySQL 런타임**으로 검증(이 프로젝트의 기존 관례).
- **후보 스캔은 잠금 없이**: 매 tick 전체 PENDING에 잠금을 걸면 비싸고 경합↑ → 스캔은 싸게, 잠금은 발행 직전 행별로(claim). 단일 폴러면 SKIP LOCKED가 no-op처럼 동작하고, 스케일아웃해야 값을 함.

### 런타임 검증 (MySQL, 2026-06-08)
- **Flyway V10 + validate 통과**: `docker compose up -d`(기존 컨테이너) → bootRun. 앱 기동 성공 = "V10 SQL ↔ 엔티티(`nextAttemptAt`) 일치". `flyway_schema_history` V10 `success=1`, `outbox_event`에 `next_attempt_at datetime(6) NULL` + `idx_outbox_dispatch(status, next_attempt_at)` 확인(구 `idx_outbox_status` 제거됨).
- **백오프 + 데드레터 end-to-end**(실패 주입): `event_type=PAYMENT_COMPLETED`·`payload='not-json-broken'`인 PENDING 행을 SQL로 직접 INSERT → 폴러가 집어 핸들러가 파싱 실패(`IllegalStateException`)로 던짐 → **retry_count 1→2→3→4→5**로 누적, `next_attempt_at`이 미래로 밀리는 동안 **그 시각 전엔 재시도 안 함**(retry 4 구간에서 `next_in_s` 6→3→0 카운트다운 후 0에서 retry 5·**FAILED** 전환을 관찰) → MAX(5) 초과로 **데드레터(FAILED)**. `last_error` HEX 디코딩 = "이벤트 페이로드 역직렬화 실패"(핸들러 메시지 = 발행 경로 `findDispatchable`→`findPendingForUpdate`(SKIP LOCKED)→publish가 MySQL에서 정상 동작했다는 증거). 검증 후 주입 행 삭제(outbox 0건, notification_log 0).
- ⚠️ 콘솔 메모: 쿼리 안 한글 LIKE 패턴(`%역직렬화%`)이 PowerShell→docker stdin 전송 중 코드페이지로 깨져 매칭 0(오탐) → DB 값은 정상 UTF-8(HEX로 확인). 한글 비교는 LIKE보다 **HEX/CHAR_LENGTH로 검증**.

## 2026-06-09 (8일차) — 다중 PG MPG-3: 정산 PG별 수수료율

> MPG-1이 PG를 2개(토스/카카오)로 늘렸으니, 다중 PG가 **실제로 값을 하는 지점** = 정산까지 끌고 온다. PG마다 수수료율이 다르므로 "매출 ≠ 결제액"의 그 차이가 **PG 선택에 따라 갈린다**. 같은 10,000원도 토스로 받으면 250원, 카카오로 받으면 280원 — 이걸 정산 계산·기록·집계에 반영했다.

### 결정 (사용자와 합의)
- **요율 출처 = `SettlementPolicy` 상수 Map**(vs application.yml 설정구동): 지금 구조(static util)를 그대로 유지하고 와이어링 추가 0. Policy 클래스 주석이 예고하던 "요율표"를 최소로 구현. 실무는 설정값/요율표 테이블이지만 학습 단계는 한 걸음씩 — 추후 config/DB로 승격해도 호출부(`calculateFee`/`rateFor`)는 그대로.
- **저장 = provider + feeRate 스냅샷**(vs provider만): 요율이 나중에 바뀌어도 "그 정산은 그때 몇 %로 뗐나"가 보존된다(OrderItem.orderPrice 스냅샷과 동일한 불변 이력 사고). fee를 fee/gross로 역산하면 반올림 오차가 있어, 적용 요율을 1급 컬럼으로.
- **feeRate = double**: 돈(원)이 아니라 **비율**이고 합산하지 않으므로 floating-point 오차 우려 없음. money는 여전히 bigint(long) 유지(가격=long 원칙 불변).
- **폴백 요율 = 3.0%**(요율표 미등록 PG): 보수적으로 가장 높게 둬서 신규 PG 요율 누락 시 과소청구 방지.

### 추가/변경
- **`SettlementPolicy`**: `FEE_RATE` 단일 상수 → **`Map<String,Double> FEE_RATES`**(TOSS 0.025·KAKAOPAY 0.028) + `DEFAULT_FEE_RATE`(0.030). `rateFor(provider)`(null/blank/미등록→폴백, 대문자 정규화) + `calculateFee(provider, gross)`(provider 인자 추가).

- **`SettlementService.run()`**: 결제의 `provider`로 요율 적용 + 항목에 스냅샷. **PG별 누적**(`LinkedHashMap`+사설 `ProviderAccumulator`)으로 `byProvider` 집계 생성.
- **DTO**: `SettlementRunResponse`에 **`List<ProviderBreakdown>`**(provider·feeRate·count·gross/fee/net) 중첩 레코드 추가, `SettlementResponse`에 provider·feeRate 노출.
- **Flyway V12**(`add_settlement_provider_fee_rate`): `settlement_entry`에 `provider varchar(30) NOT NULL DEFAULT 'TOSS'`·`fee_rate double NOT NULL DEFAULT 0.025`(기존 행=단일 2.5% TOSS 세계 백필).
- 테스트: `SettlementServiceTest` **+2**(PG별 요율 차이+PG별 분해 검증 / 미등록 PG 폴백 요율) + 기존 run 테스트에 provider·feeRate 단언 추가, `scheduled(...)` 호출부(SettlementServiceTest·ReconciliationServiceTest) 새 시그니처로 갱신 → **전체 138개**(136→138).

## 2026-06-09 (8일차) — 다중 PG MPG-2: 대사 PG별 분류/표시 강화

> MPG-1(라우터)·MPG-3(정산 PG별 수수료)에 이어, **대사(reconciliation)도 PG를 1급으로** 다룬다. 불일치가 어느 PG의 거래인지 분류·표시·필터해, 다중 PG 운영에서 "어느 PG에서 어긋남이 몰리는가"를 본다.

### 핵심 함정 (설계 근거)
- **거래ID 프리픽스 ≠ provider**: 카카오 어댑터는 거래ID 프리픽스가 `KAKAO-`인데 provider는 `KAKAOPAY`(MPG-1 결정). 그래서 불일치의 PG를 거래키 문자열에서 파싱하면 **틀린다** → PG가 정산 리포트에 **자기 provider를 직접 실어** 알려주게 했다(`PgSettlementRecord.provider`). 우리 쪽은 MPG-3에서 `SettlementEntry.provider`를 이미 보유 → 거래키마다 provider를 양쪽에서 확실히 얻는다.

### 추가/변경
- **`PgSettlementRecord`**에 `provider` 필드 추가. `AbstractMockPaymentGateway`가 승인/환불 시 원장에 `provider()`를 함께 기록(공통 베이스라 토스·카카오 자동 반영).
- **`Mismatch`**에 `provider`(varchar30) 필드 + `Mismatch.of(...)` 시그니처에 provider 추가. **Flyway V13**(`mismatch.provider`, DEFAULT 'TOSS' 백필).
- **`ReconciliationService.reconcile()`**: 거래키별 `provider = (우리 있으면) SettlementEntry.provider : PgSettlementRecord.provider`로 도출 → `Mismatch`에 저장 + **PG별 분해 집계**(`TreeMap`+사설 `ProviderAccumulator`로 알파벳순). 5분류 로직 자체는 불변.
- **`ReconciliationResult`**에 중첩 `ProviderReconciliation`(provider별 matched/4분류/total/alreadyHandled) + `byProvider` 리스트.
- **`MismatchResponse`**에 provider. **불일치 목록 provider 필터**: `MismatchRepository.findByProvider`·`findByStatusAndProvider`, `getMismatches(status, provider, pageable)`(둘 다 선택·provider 대문자 정규화·4분기), 컨트롤러 `@RequestParam(required=false) String provider`.
- 테스트: `ReconciliationServiceTest` **+5**(PG별 분해 / MISSING_IN_OURS provider=PG통보값(프리픽스 아님) / 필터 provider만·status+provider·필터없음) + 기존 헬퍼(`our`·`pg`)에 provider 오버로드, `Mismatch.of` 호출부 갱신 → **전체 143개**(138→143).

### 결정
- **PG 식별 = 리포트가 직접 싣는다**(프리픽스 파싱 X): provider≠프리픽스라 파싱은 깨짐. PG 리포트가 자기 출처를 밝히는 게 현실적이기도(실제 PG 정산 파일도 PG가 누군지 자명).
- **byProvider는 알파벳순(TreeMap)**: 거래키 집합이 HashSet이라 순회 순서가 비결정적 → 출력/테스트 안정을 위해 provider로 정렬.
- **provider 필터는 파생 쿼리 4분기**(QueryDSL 아님): 같은 리포의 기존 `findByStatus`(파생 쿼리)와 일관. 선택 필터 2개라 조합 4개를 서비스에서 분기(단순·가독성).

## 2026-06-10 (9일차) — 다중 PG 비용기반 라우팅 + 요율 출처 단일화

> 다중 PG 라우팅의 마지막 전략. 클라이언트가 `provider="AUTO"`로 두면 **가장 싼 PG로 자동 라우팅**. 그 과정에서 요율의 출처를 **PG 게이트웨이로 통합**(라우팅 비용 = 정산 수수료, 한 곳에서 정의).

### 결정 (요율 출처 — 사용자와 합의)
- **요율은 PG 고유 속성 → 게이트웨이가 단일 출처**(`PaymentGateway.feeRate()`). 기존엔 `SettlementPolicy`(정산 패키지)에 요율 Map이 있었는데, 라우터(결제 패키지)가 비용기반 선택을 하려면 요율이 필요 → 라우터→SettlementPolicy는 **의존 역방향**이라 못 씀. 그래서 요율을 게이트웨이로 올리고, **정산이 라우터에서 요율을 읽도록**(settlement→payment 정방향) 뒤집음. 중복 0·드리프트 위험 제거.
- **AUTO = 비용기반 사전값**: 명시 PG도 그대로 지원하되, "AUTO"면 라우터가 최저 요율 PG 선택. 페일오버 순서도 비용 오름차순으로 바꿔, AUTO든 명시든 **더 싼 대체 PG로 먼저** 넘어감.

### 추가/변경
- **`PaymentGateway.feeRate()`**(포트에 추가) — `TossMockGateway` 0.025·`KakaoPayMockGateway` 0.028. 요율 단일 출처.
- **`PaymentGatewayRouter`**: `feeRateOf(provider)`(미등록/blank→`DEFAULT_FEE_RATE` 3.0% 폴백), `cheapestProvider()`(최저 요율·동률 알파벳순), **`approveWithFailover`가 `provider="AUTO"`면 최저가 1차 선택**, `failoverOrder`를 **비용 오름차순**(동률 알파벳)으로. `AUTO`·`DEFAULT_FEE_RATE` 상수.
- **`SettlementPolicy`**: 요율 Map·`rateFor`·`DEFAULT_FEE_RATE` **제거** → `calculateFee(double feeRate, long gross)`(요율을 받음)+`PAYOUT_DELAY_DAYS`만 남김.
- **`SettlementService`**: `PaymentGatewayRouter` 주입 → `feeRateOf(provider)`로 요율 조달 후 `calculateFee(rate, amount)`.
- **`PaymentRequest`** 스키마에 AUTO 명시. **FE 결제화면**에 "자동 (최저 수수료)" 옵션(`provider="AUTO"`).
- 테스트: `PaymentGatewayRouterTest` **+3**(feeRateOf·AUTO 최저가[3번째 더 싼 fake로 최소값 선택 증명]·비용 오름차순 페일오버), `SettlementServiceTest`는 `paymentGatewayRouter.feeRateOf` 목으로 갱신 → **전체 152개**(149→152). FE `tsc`/`next lint` 클린.
